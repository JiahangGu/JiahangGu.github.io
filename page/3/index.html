MIT6824-Labs整体总结

发表于 2022-02-19 分类于 实验笔记
本文字数： 739 阅读时长 ≈ 1 分钟

收获：

  1. 写好测试真的是非常有必要，在每次大幅度修改后，均可以通过测试来找出对此前代码的功能正确性的影响部分。
  2. 多并发编程需要关注的点太多。总体来说，并发编程的经验相对之前几乎为0来说，有了很明显的提升吧。日后再实现并发编程功能，在开始会有一些需要重点关注的问题的轮廓，在具体实现会注意并发的竞态问题。
    1. 需要为多线程同时访问的数据加锁
    2. 要考虑加锁的范围，在存在较长等待时间的情况下应该复制一个临时数据出来，并在其上操作，避免加锁时间过久导致
    3. 在某段加锁的代码中，后续需要使用公有数据，但不可以加锁，同样复制一个临时数据
    4. 调试无法通过单点实现，需要尽可能增加详细的日志输出，通过日志梳理表现不正确的行为，进而确定具体存在的逻辑问题
  3. raft部分实现细节很多，往往某一个细节存在纰漏就会导致整个系统出问题，应该尽可能确定raft论文中的细节。
  4. 快照是用来帮助应用程序恢复状态的，在raft中只需要记录处理的log，当程序挂掉后，可以加载快照数据到内存，并根据log对内存数据更新，从而恢复到崩溃之前的状态。快照数据通过raft复制到所有的服务器上，并且对log压缩。
  5. 最后是对项目的整体架构的学习。在最初实现时，会用一个函数代替某些想要做到的功能，然后取实现函数，而不是在一个代码段中堆积。其次在实现后，会主动封装重用的代码块，避免复制粘贴。但自己写的函数往往没有测试，依赖于lab中提供的测试。
  6. 对golang的使用更加熟悉，例如并发，通道等。-race参数非常好用啊。

不足：

  1. 并发存在的bug较难解决和复现，尤其是lab4，还有一定概率运行失败。留待以后换一种心境再实现一遍。
  2. raft的细节太多，初始查文章资料的时候不知道重点在哪，对于一些用处也没完全理解。对于实际的运行逻辑和处理细节考虑的不够周到。

MIT6824-Lab4总结

发表于 2022-02-19 分类于 实验笔记
本文字数： 6.6k 阅读时长 ≈ 6 分钟

加油

任务描述

  1. 建一个分片的k-v存储系统。提高吞吐量
  2. 存储的两个重要成分：
    1. 复制组的集合，每个复制组负责所有分片的一个子集，使用raft复制组内的分片数据
    2. 分配控制器，决定每个复制组应该管理哪些分片（配置），配置改变时要保存历史记录。客户端查询控制器找到key对用的复制组，复制组查询控制器找到应该负责哪些分片。控制器也应该用raft复制
  3. 本实验主要难点是解决配置，因为配置会在将分片分配给组时变化。单个组内的所有成员必须统一配置变化和查询的顺序。
  4. 总体架构类似于Flat Datacenter Storage, BigTable, Spanner, FAWN, Apache HBase, Rosebud, Spinnaker，都是一个配置服务和一组复制组的集合

Task1

控制器管理者很多版本的配置，当配置改变时，控制器建一个新的。客户端和服务器可以查询特定版本的配置。

人月神话-持续更新

发表于 2022-02-18 分类于 读书笔记
本文字数： 522 阅读时长 ≈ 1 分钟

第一遍

整本书读下来，有很多经典的结论，但结合我目前的项目经历，真正让我觉得对自己帮助最大的两句话是：
将做事方式调整到追求完美是学习编程的最困难部分。

人们通常期望项目在接近结束时，软件项目能收敛得快一些，然而，情况却是越接近完成，收敛得越慢。

也是我自主学习中遇到的困难。一是在目标大概完成（比如题目通过，项目可以正常运行等）后得到了较强的自我认同感和满足感，进而不再继续尝试优化和改进，并逐渐觉得这样的方式也无可厚非。二是在做项目时，由于开发过程不规范，即在完成后实现测试，编写测试用例时未能真正的考虑全面case，会希望能够尽快完成自己设定的目标，但目标不一定合理。

实习完的这段时间，有对自己技能和态度的一些反思，进而进一步考虑如何更好的完善职业技能。有一定的进步，希望在后续的时间能做到更好。
